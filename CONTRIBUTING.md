# Contributing to Groovy Bugs

Thank you for your interest in contributing to Groovy Bugs! We want to make contributing to this project as easy and transparent as possible.

## 📋 Code of Conduct
By participating in this project, you agree to treat everyone with respect and kindness.

## 🛠️ Development Process

1. **Fork and clone** the repository.
2. **Follow the setup instructions** in `README.md` to get your local environment running.
3. **Create a feature branch:** `git checkout -b feature/your-feature-name` or `bugfix/issue-description`.
4. **Make your changes.** Be sure to test them locally on both frontend and backend.
5. **Commit your changes:** Use clear, descriptive commit messages.
6. **Push to your fork:** `git push origin your-branch-name`.
7. **Submit a Pull Request (PR)** against the `main` branch.

## 🏗️ Architecture & Best Practices

- **Backend:** 
  - Express.js architecture divided into `models`, `routes`, `middleware`, and `services/scripts`.
  - All new routes accessing protected data must use `import { requireAuth } from '../middleware/auth.js'`.
  - Admin-only mutation routes must use `import { requireAuth, requireAdmin } from '../middleware/auth.js'`.
  - Keep business logic inside controllers or route handlers; do not put it in the React frontend.
  - Never commit hardcoded secrets or environment variables.

- **Frontend:**
  - React application generated by Vite.
  - State management uses React Context (`CartContext`, `ProductContext`).
  - Styling is done via TailwindCSS. Prefer utility classes over custom CSS where possible.
  - Use `src/services/api.js` for all backend communication to ensure base URLs and headers are handled correctly.

## 🐞 Bug Reports

If you discover a bug, please search existing issues first. If there isn't one, open a new issue and include:
- A clear, descriptive title.
- Steps to reproduce the issue.
- Expected behavior vs. actual behavior.
- Screenshots if it's a UI issue.
- Information about your environment (Browser, Node version).
